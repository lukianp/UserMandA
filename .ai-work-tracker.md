```powershell
# Strategy 3: Microsoft Graph API Bridge (cross-platform auth bridge)
try {
    Write-ModuleLog -ModuleName "AzureDiscovery" -Message "Attempting Microsoft Graph API bridge authentication..." -Level "INFO"

    if (Get-Module -ListAvailable -Name Microsoft.Graph.Authentication) {
        Import-Module Microsoft.Graph.Authentication -ErrorAction Stop

        if ($Configuration.TenantId) {
            Connect-MgGraph -TenantId $Configuration.TenantId -Scopes "https://management.azure.com/.default" -UseDeviceCode -ErrorAction Stop | Out-Null
        } else {
            Connect-MgGraph -Scopes "https://management.azure.com/.default" -UseDeviceCode -ErrorAction Stop | Out-Null
        }

        $graphToken = Get-MgContext | Select-Object -ExpandProperty AccessToken
        if ($graphToken) {
            $accessToken = ConvertTo-SecureString $graphToken -AsPlainText -Force
            $accountId = (Get-MgContext).Account
            $tenantId = (Get-MgContext).TenantId
            $context = Connect-AzAccount -AccessToken $accessToken -AccountId $accountId -TenantId $tenantId -WarningAction SilentlyContinue -ErrorAction Stop
            Write-ModuleLog -ModuleName "AzureDiscovery" -Message "Graph API bridge authentication successful" -Level "SUCCESS"
            return $context
        }
    }
} catch {
    Write-ModuleLog -ModuleName "AzureDiscovery" -Message "Graph API bridge authentication failed: $($_.Exception.Message)" -Level "WARN"
}
```

### Technical Context

**What Was Implemented:**
Multi-strategy authentication fallback pattern to resolve Azure PowerShell MSAL library memory corruption (System.AccessViolationException). The Service Principal authentication fails with memory corruption, so the system now automatically falls back to alternative authentication methods that work (particularly Graph API Bridge and Device Code).

**Root Cause:**
MSAL token cache corruption in Azure PowerShell modules when using Service Principal with Client Secret credentials. The authentication session becomes corrupted, causing memory access violations.

**Solution Approach:**
Implemented 6-strategy fallback with headless-first priority:
1. Try Service Principal (expected to fail with memory corruption - caught specifically)
2. Try Azure CLI if already authenticated
3. Try Microsoft Graph API Bridge (get Graph token, bridge to Azure) - **NEW**
4. Try Managed Identity if on Azure
5. Try Device Code (headless interactive - displays code for browser auth) - **NEW**
6. Try Interactive Browser (last resort - requires GUI)

**Key Implementation Details:**
- Specific exception handler for `System.AccessViolationException` to detect memory corruption
- Graph API Bridge strategy uses `Connect-MgGraph` with device code, then extracts token for Azure
- Device Code strategy allows authentication in headless environments (SSH, containers, etc.)
- All strategies log detailed messages for troubleshooting
- Failure of one strategy automatically triggers next strategy
